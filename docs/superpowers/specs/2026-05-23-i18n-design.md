# i18n (Traditional Chinese + English) Design

**Status:** Draft for review — 2026-05-23

## Goal

Make every user-facing string in the dsvisual app shell switchable between Traditional Chinese (zh-Hant) and English at runtime, with the preference persisted across reloads, and with the architecture extensible to a third language without re-plumbing.

## Background

Today the project is in a half-translated state:

- `app.js` is entirely English — `METHOD_GROUPS` titles, control button labels, status text, tutorial walkthrough, slide-progress counters all hard-coded in English. No CJK characters appear anywhere in the file.
- `index.html` declares `<html lang="zh-Hant">` but a few labels show both languages concatenated (`中 / EN`, `設定 / Settings`, `程式碼面板行距 / Code panel line-height`, `預覽 / Preview`) — a poor-man's bilingual workaround.
- Slide content is already truly bilingual: `slides_db.js` stores `{ zh, en }` value objects, and `build_slides.js`'s `pick(value, lang)` helper resolves them at render time. An existing button `[data-testid="slide-lang-toggle"]` toggles the slide language only.

The path forward: extend the `{ zh, en }` lookup pattern from the slide pipeline to the whole app shell, promote the existing slide-language button to an app-wide language toggle, and audit/translate the strings.

## Scope

### In scope (translated)

- Nav bar — category pills, hover-dropdown method names, sub-tab buttons (Creational/Structural/Behavioral/Architectural).
- Currently-active method section heading.
- Control buttons — `Push()`, `Pop()`, `Enqueue()`, `Dequeue()`, `Step`, `Run`, `Reset`, `Add Edge`, `Clear`, `Build`, copy-to-clipboard `Copied`/`Copy`, etc.
- Status & toast messages — `Switched to <mode>`, `✓ Copied`, heap tutorial walkthrough text (`Completed`, the full summary string, step counter).
- Slide viewer chrome — `Slide N / M` progress label, viewer title, close button aria-label.
- Settings drawer — title, slider labels, preview header.
- All meaningful `aria-label` attributes.

### Out of scope (stays English)

- Code panel `.cpp` filenames (they are real filenames in the repo).
- Visualizer-internal SVG/canvas text — node values, "root", "head", in-canvas instructions, hash table cell labels. The visualizer renderers are spread across 50+ methods and rewriting them is a separate, much larger project.
- Slide *content* — already i18n via `slides_db.js`. Only the chrome around it gets the new treatment.
- The C++ source files themselves.

### Non-goals

- RTL languages — none of the in-scope languages need it.
- Pluralization rules — current strings have no plurals worth special-casing.
- Date / number / currency formatting — no such strings exist in the app.

## Architecture

### Module: `i18n.js`

New file. Loaded as a regular `<script>` *before* `app.js` in `index.html`, so first paint uses the right language.

Public API (module-scope globals via a namespace object, matching the project's no-build-step convention):

```js
window.I18N = {
    SUPPORTED: ['zh', 'en'],
    getCurrentLanguage(),                   // returns 'zh' or 'en'
    setLanguage(lang),                      // mutates + persists + re-renders
    t(key, vars?),                          // returns translated string with {var} substitutions
    applyTranslations(root = document),     // walk [data-i18n-key], set textContent + attrs
};

// Ergonomic global alias for the most-called helper; `t(...)` reads cleaner inline.
window.t = window.I18N.t;
```

Module-scope state:

```js
const STORAGE_KEY = 'dsvisual-lang';
let currentLang = detectInitialLanguage();   // runs at script load
document.documentElement.lang = currentLang === 'zh' ? 'zh-Hant' : 'en';
```

### Translation table shape

```js
const TRANSLATIONS = {
    zh: {
        'group.linear':                 '線性結構',
        'group.trees':                  '樹',
        'group.graphs':                 '圖',
        'group.hash':                   '雜湊與機率資料結構',
        'group.heaps':                  '堆積 / 優先佇列',
        'group.sorting':                '排序',
        'group.search':                 '搜尋與字串比對',
        'group.oop':                    'OOP 概念',
        'group.patterns':               '設計模式',
        'group.patterns-creational':    '建立型',
        'group.patterns-structural':    '結構型',
        'group.patterns-behavioral':    '行為型',
        'group.patterns-architectural': '架構型',

        'method.stack-array':           '堆疊（陣列實作）',
        'method.queue':                 '佇列',
        // ...all ~50 method IDs

        'btn.push':                     'Push()',     // some keep English
        'btn.pop':                      'Pop()',
        'btn.step':                     '單步',
        'btn.run':                      '執行',
        'btn.reset':                    '重置',
        'btn.copy':                     '複製',
        'btn.copied':                   '✓ 已複製',

        'settings.title':               '設定',
        'settings.density':             '程式碼面板行距',
        'settings.preview':             '預覽',

        'slide.progress':               '第 {n} 張 / 共 {total} 張',
        'slide.toggle-aria':            '切換語言',

        'status.switched-to':           '已切換到 {mode}',
        'tutorial.completed':           '已完成',
        // ...
    },
    en: {
        'group.linear':                 'Linear Structures',
        // ...mirror every zh key
    },
};
```

**Key naming rules:**
- `group.<group-id>` — the existing `group.id` from `METHOD_GROUPS` (e.g., `linear`, `trees`, `patterns-creational`). Lookup is `t('group.' + group.id)`.
- `method.<method-id>` — the existing `method.id` (e.g., `stack-array`, `tree-bst`). Lookup is `t('method.' + method.id)`.
- `btn.<action>`, `settings.<key>`, `slide.<key>`, `status.<event>`, `tutorial.<event>`, `aria.<context>` for chrome.

Dot-notation keeps related strings grouped and makes audits trivial (`grep "'btn\."`).

### `t(key, vars?)`

```js
function t(key, vars) {
    const table = TRANSLATIONS[currentLang] || TRANSLATIONS.en;
    let str = table[key];
    if (str == null) {
        console.warn('[i18n] missing key:', key, 'for lang:', currentLang);
        str = TRANSLATIONS.en[key] || key;
    }
    if (vars) {
        str = str.replace(/\{(\w+)\}/g, (_, name) =>
            vars[name] != null ? String(vars[name]) : '{' + name + '}');
    }
    return str;
}
```

Fallback chain: current language → English → raw key. Missing keys log a warning so they're noticed in development.

Substitution: `{name}` placeholders replaced from the `vars` object. Enough for `slide.progress` and `status.switched-to`; no further templating needed.

## Switcher & persistence

### Repurposing the existing button

The existing `[data-testid="slide-lang-toggle"]` button becomes the app-wide language toggle. Changes:

- Rename `data-testid` to `lang-toggle` (search-replace across tests).
- Label shows the *target* language using the "switch to X" idiom — text reads `中` when current is `en`, `EN` when current is `zh`. Language-neutral, no translation key needed for the button label itself.
- Click handler:
    ```js
    btn.addEventListener('click', () => {
        const next = I18N.getCurrentLanguage() === 'zh' ? 'en' : 'zh';
        I18N.setLanguage(next);
    });
    ```

### Boot sequence (in `i18n.js`, runs at script-load time)

```js
function detectInitialLanguage() {
    const saved = localStorage.getItem(STORAGE_KEY);
    if (SUPPORTED.includes(saved)) return saved;
    const nav = (navigator.language || '').toLowerCase();
    return nav.startsWith('zh') ? 'zh' : 'en';
}
```

`localStorage['dsvisual-lang']` values are `'zh'` or `'en'` only — never the full locale codes. The `<html lang>` attribute uses the BCP47 forms `zh-Hant` / `en`.

### `setLanguage(lang)` ordering

1. Guard: ignore if not in `SUPPORTED` or already current.
2. Update `currentLang`.
3. `localStorage.setItem(STORAGE_KEY, lang)`.
4. `document.documentElement.lang = lang === 'zh' ? 'zh-Hant' : 'en'`.
5. Update the toggle button label.
6. Re-apply translations to all `[data-i18n-key]` static nodes (Layer 1 below).
7. Dispatch `new CustomEvent('languagechange', { detail: { lang } })` on `document` (Layer 2 below).

Step 6 runs *before* step 7 so that by the time dynamic re-renders fire, the surrounding static chrome is already in the new language — no flash of mixed state.

## Re-render mechanism

Two complementary layers; both driven by `setLanguage()`.

### Layer 1 — static HTML chrome (declarative)

Translatable text and attributes in `index.html` get annotated:

```html
<h2 id="settings-drawer-title" data-i18n-key="settings.title">Settings</h2>

<label for="code-density-slider" data-i18n-key="settings.density">
    Code panel line-height
</label>

<button data-testid="lang-toggle"
        data-i18n-aria-label="slide.toggle-aria">EN</button>
```

A single walker function refreshes them:

```js
function applyTranslations(root) {
    root.querySelectorAll('[data-i18n-key]').forEach((el) => {
        el.textContent = t(el.dataset.i18nKey);
    });
    root.querySelectorAll('[data-i18n-aria-label]').forEach((el) => {
        el.setAttribute('aria-label', t(el.dataset.i18nAriaLabel));
    });
    // Similar passes for data-i18n-placeholder, data-i18n-title if any appear.
}
```

Called once at boot after DOM ready, and again inside `setLanguage()`.

The initial English content in HTML is just a sane fallback if JS is disabled or if a key is missing.

### Layer 2 — dynamically built UI (imperative)

Anything `app.js` constructs at render time calls `t(...)` directly (the global alias from above). On language change, listeners re-run:

```js
document.addEventListener('languagechange', () => {
    renderCategoryNav();              // nav pills + dropdowns
    updateActiveMethodHeading();      // current method's title in the section
    // switchMode(currentMode) re-runs the per-mode label setup
    // (Push/Pop, Enqueue/Dequeue, Add Edge, etc.)
    switchMode(currentMode);
    // If the slide viewer is currently open, redraw its chrome + slide body
    rerenderSlideViewerIfOpen();
});
```

Each listener calls existing render paths — no new state machine. The exact slide-viewer hook depends on what the existing viewer code exposes; the implementation plan resolves the precise name. Short-lived status toasts (`✓ Copied`, `Switched to ...`) are not updated in-flight; the *next* toast uses the new language.

### Which strings live in which layer

| Element                                  | Layer | Why |
| ---------------------------------------- | ----- | --- |
| Settings drawer title and labels         | 1     | Static HTML |
| Slide viewer modal title, close aria     | 1     | Static HTML |
| Density preview header                   | 1     | Static HTML |
| Lang-toggle aria-label                   | 1     | Static HTML |
| Nav pills, dropdown methods, sub-tabs    | 2     | Built by `renderCategoryNav` |
| Method section heading                   | 2     | Set by `renderMethodSections` / `switchMode` |
| Control buttons (Push/Pop/Add Edge/...)  | 2     | Set by mode-specific code in `switchMode` |
| Slide progress `Slide N / M`             | 2     | Set on each slide change |
| Status messages and tutorial text        | 2     | Set on event |

## Test strategy

### Existing tests — pin language to `en`

```js
test.beforeEach(async ({ page }) => {
    await page.addInitScript(() => localStorage.setItem('dsvisual-lang', 'en'));
    // ... existing setup
});
```

`addInitScript` runs *before* any page script on every navigation, so `i18n.js`'s `detectInitialLanguage()` reads `'en'` immediately. The existing Playwright tests across the 5 spec files (currently 90 tests, plus 44 node unit tests that don't touch the DOM) keep their English assertions unchanged.

### New `tests/i18n.spec.js`

1. **Toggle flips nav** — load page in `en`, assert a known pill reads `Linear Structures`; click `[data-testid="lang-toggle"]`; assert same pill now reads `線性結構`.
2. **Persistence across reload** — toggle to zh, reload page (no `addInitScript`), assert pill reads `線性結構` and `localStorage['dsvisual-lang']` is `'zh'`.
3. **`<html lang>` updates** — assert `documentElement.lang === 'en'`; toggle; assert it is now `'zh-Hant'`.
4. **`data-i18n-key` walker correctness** — toggle, then assert settings drawer title reads `設定`; toggle back, assert `Settings`.
5. **Browser auto-detect (zh)** — `browser.newContext({ locale: 'zh-TW' })`, navigate fresh (no localStorage), assert pill reads `線性結構`.
6. **Browser auto-detect (en)** — `browser.newContext({ locale: 'en-US' })`, navigate fresh, assert pill reads `Linear Structures`.
7. **Missing-key fallback** — call `I18N.t('definitely.missing.key')` via `page.evaluate`, assert it returns the raw key and that a console warning was emitted.

### Slide-viewer test (`tests/slides_viewer.spec.js`)

The button no longer reads `中 / EN`; it reads `中` or `EN` depending on the *current* language. Replace any text-based assertion on the button with a testid-based assertion, and assert slide content language switches when the toggle is clicked.

## Translation review process

The implementation plan will list every English string with its proposed Chinese translation in a single table at the top of one task. The user signs off on the translation table before the bulk edits land. CS terms are domain-sensitive (e.g., "Disjoint Set" could be `互斥集合` or `並查集`) — seeing the full list at once prevents drift across files.

## Risks & limitations

- **Visual layout shifts** — zh strings often differ in width from en. Current Playwright tests don't cover visual regressions; we accept minor shifts and fix only if a layout breaks.
- **Translation accuracy** — CS terminology in Traditional Chinese is not universal; the review table above is the mitigation.
- **`navigator.language` detection brittleness** — only kicks in on first visit (no `localStorage`). Tests pin localStorage to make it deterministic; auto-detect tests use explicit `locale` contexts.
- **Mixed-language flicker risk** — the ordering rule in `setLanguage()` (static walker before event dispatch) is the mitigation.

None — all design choices confirmed in the brainstorming session (scope, coupling, storage shape, default-language detection).
