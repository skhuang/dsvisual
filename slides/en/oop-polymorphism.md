## Polymorphism (Virtual Functions)

Polymorphism lets objects of different types be operated on through a common interface. In C++, `virtual` functions together with the vtable mechanism implement runtime dynamic dispatch.

---

## Core Concept

Every class with `virtual` functions has a Virtual Method Table (vtable) storing the actual addresses of its virtual functions. Each object carries a hidden vptr pointing to its class's vtable.

- `virtual` function: declared in the base class; derived classes may override it with `override`.
- `= 0` (pure virtual): makes the base class abstract, forcing all derived classes to implement it.
- Dynamic dispatch: `ptr->draw()` follows vptr to vtable to the correct function address at runtime.
- Static dispatch (non-`virtual`): function address resolved at compile time, no indirection.

---

## Operation Flow

1. Declare abstract base class `Shape` with pure virtual `draw()` and `area()`.
2. `Circle` and `Rectangle` inherit `Shape` and implement their own versions with `override`.
3. Store different objects through `Shape*` pointers; the loop calls `shape->draw()`, dispatching automatically.
4. vtable lookup happens at runtime; each virtual call adds two pointer indirections overhead (read vptr, then read the vtable slot).

> Each object has one vptr; each class has one vtable. The vtable is built by the compiler at compile time and stored in read-only data. At runtime only pointer indirection occurs.

## Static vs Dynamic Dispatch

| Property | Static Dispatch (non-virtual) | Dynamic Dispatch (virtual) |
| --- | --- | --- |
| Resolution time | Compile time | Runtime |
| Overhead | None | Two pointer indirections |
| Polymorphism | No | Yes |
| Abstract class | N/A | Declared with `= 0` |

$$T_{\text{virtual}} = T_{\text{direct}} + 2 \cdot T_{\text{ptr-deref}}$$

Virtual call overhead: one vptr access + one vtable lookup = two pointer indirections over a direct call.

## Source Code

```cpp
class Shape {
public:
    virtual ~Shape() {}
    virtual void draw() const = 0;   // pure virtual
    virtual double area() const = 0; // pure virtual
};

class Circle : public Shape {
    double radius;
public:
    explicit Circle(double r) : radius(r) {}
    void draw() const override {
        cout << "Drawing Circle(" << radius << ")" << endl;
    }
    double area() const override {
        return 3.14159 * radius * radius;
    }
};

int main() {
    vector<Shape*> shapes;
    shapes.push_back(new Circle(5.0));
    shapes.push_back(new Circle(3.0));

    for (const auto* s : shapes) {
        s->draw();  // dynamic dispatch via vtable
        cout << "Area: " << s->area() << endl;
    }
    for (auto* s : shapes) delete s;
}
```

## Pros, Cons & When to Use

- Pro: Open/Closed Principle — add new derived classes without modifying existing code.
- Pro: uniform interface — operate on heterogeneous collections through base pointers/references.
- Con: vptr indirection may inhibit inlining, impacting performance on hot paths.
- Con: each object grows by one vptr (typically 8 bytes); each class has one vtable.
- Use when a unified interface must handle different types, e.g. GUI widget rendering, game entity updates.

---

## Summary

- `virtual` functions + vtable implement runtime dispatch: vptr → vtable → actual function.
- Pure virtual (`= 0`) makes the class an abstract interface, mandating implementation in derived classes.
- Virtual call overhead is just two pointer indirections — negligible in most scenarios.
- Polymorphism underpins many design patterns such as Strategy and Template Method.
