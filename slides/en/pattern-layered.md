## Layered Architecture

Layered architecture organizes a system into horizontal layers (Presentation, Business, Data); each layer uses only the layer directly below it to separate concerns.

---

## Core Concept

Strict downward dependency — a layer never calls upward or skips a layer. Each layer knows only its immediate neighbor below, keeping layers independent.

- Each layer has a single responsibility with a clear focus.
- Lower layers are unaware of higher ones, keeping coupling low.
- Swapping a layer's implementation doesn't ripple upward to other layers.

---

## Operation Flow

1. A request enters at the Presentation layer, which handles formatting and display.
2. The Presentation layer calls the Business layer, which applies rules and validates data.
3. The Business layer calls the Data layer to retrieve raw data; results flow back up through each layer.

## Trade-offs & When to Use

| Aspect | Notes |
| --- | --- |
| Separation of concerns & maintainability | Each layer has a clear role, making maintenance straightforward |
| Per-layer testability | Each layer can be tested in isolation by mocking the layer below |
| Cost | Requests are passed layer-to-layer; risk of pass-through sinkhole layers |

---

## Source Code

```cpp
// Business layer — applies rules; calls only the layer below.
class BusinessLayer {
    DataLayer data;
public:
    string process() {
        return "[validated] " + data.fetch();
    }
};

// Presentation layer — formats output; calls only the layer below.
class PresentationLayer {
    BusinessLayer business;
public:
    void show() {
        cout << "Display: " << business.process() << endl;
    }
};
```

---

## Pros, Cons & When to Use

- Pro: clear structure; each layer can be independently maintained and tested.
- Pro: swapping a layer's implementation (e.g., switching databases) does not affect upper layers.
- Con: extra indirection as requests pass through each layer.
- Con: risk of pass-through sinkhole layers when intermediate layers add no real logic.
- Use for business applications with clear horizontal concerns (e.g., classic 3-tier architecture).

---

## Summary

- Layered architecture organizes a system into horizontal layers with strict downward dependency.
- It is the classic n-tier structure, widely used in business applications and web systems.
- Clear separation of responsibilities allows each layer to be developed, tested, and replaced independently.
